# Architecture Audit（定期アーキテクチャ監査の汎用骨格）

実装が進んでから「方向性がずれていないか」「技術選定が古くなっていないか」を、実装に関与していない fresh セッションに定期的に監査させるハーネス。

**プロジェクト専用の監査スキルが存在する場合はそちらに従うこと。** このスキルは新プロジェクトへ導入するときの雛形。

## プロジェクト側が定義するもの（設定）

`<project>-architecture-audit` スキル（または監査プロンプト doc）に次を書く:

- **経緯と前提**: プロジェクトの目的・引き継ぎ経緯・設計資料の位置づけ（参考か制約か）・意図的に無視している制約。**監査者はこれを知らないと「設計書と違う」型の誤指摘を量産する**
- **読むべき現状コード**: schema / エントリポイント / ミドルウェア / 主要サービス / migrations / compose 等の代表ファイルリスト
- **技術スタックの検証対象**: 採用ライブラリと、比較対象になる代替（例: Bun/Hono/Drizzle vs Express/Fastify/Prisma）
- **レポートの出力先**: 例 `docs/audits/YYYY-MM-DD-architecture-audit.md`

## 監査の不変骨格

1. **前提の読み込み** — プロジェクト前提とコンテキストを監査プロンプトに必ず含める。記憶から推測しない。ファイルがあるものはファイルを読む
2. **実装の現状把握** — 指定された現状コードを読み、テストが走るなら走らせて pass/fail を記録する
3. **時間依存の事実は web で検証する** — ライブラリの最新版・メンテ状況・既知問題は必ず web（npm registry / GitHub releases / 公式 docs）で当日確認し、レポートに監査日を明記する。訓練データの知識で判断しない
4. **fresh セッションに委託する** — 実装した本人に監査させない。runner（`run-codex-review.sh` 等）または新しい CLI セッションで独立コンテキストを立てる
5. **レポート** — 日付入りで出力先に保存。指摘は「現状 → リスク → 推奨 → 移行コスト」の形で、緊急度を分ける。前回監査があれば差分（解消済み・新規・据え置き）を明示する

## 頻度の目安

- 大きなフェーズ境界（設計→実装、実装→リリース準備）ごとに 1 回
- 定常期は月 1 回程度。監査のたびに指摘ゼロが続くなら間隔を伸ばす（Hygiene と同じ発想）

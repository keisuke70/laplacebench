# LaplaceBench 実験軸の整理

作成: 2026-07-21。パイロット(判別力確認)完了後のブレインストームを体系化した
もの。[benchmark-strategy-ja](benchmark-strategy-ja.md) と
[design-v0.1](design-v0.1.md) を前提とし、条件は**絶対にラベルなしで混ぜない**
という既存原則の下で運用する。

## 軸1: 観測形式 (modality)

| 条件 | 内容 | 位置づけ |
|---|---|---|
| `json`(現行) | 盤面文字列 約150-250トークン/手 | **主レール。** 安価・高速・状態追跡を純粋測定 |
| `image` | 決定論レンダラーのPNG 約1,000-1,600トークン/手 | ペア診断トラック |
| `image-nolabel` | 駒の文字刻印なし(色のみ) | 色知覚の単離測定 |
| ブラウザ操作 | クリックで着手 | **別ベンチに完全分離**(ブリーフ§3.3の確定事項)。エージェント身体性枠 |

- 仮説A: 挟みの脅威・中央の形は視覚ゲシュタルトの方が見えやすく、**画像で強くなる**モデルがある
- 仮説B: 精密な座標読み取りが弱く、**画像で崩れる**モデルがある
- Vision−Stateギャップは符号すら事前に不明 → モデル間で逆転すればそれ自体が発見
- レンダラー設計: 駒に文字(R/B/Y/G)刻印で色知覚と空間理解を分離。刻印A/Bで色認識だけを単離

## 軸2: コンテキスト条件 (context)

design-v0.1 §3.4 Context Lab の拡張。

| 条件 | 内容 | 測るもの |
|---|---|---|
| `cold`(現行) | ルールブックのみ・毎試合新規 | 初見の強さ(ゼロショット) |
| `primed` | 開始時に戦略ガイド注入 | 知識利得。ガイドは自己対戦から自動生成も |
| `series-transcript` | 連戦で全文脈持ち越し | 生の蓄積能力 |
| `series-notes` | 対局後に教訓を自筆させ次戦冒頭に注入 | 蒸留・記憶設計の能力 |

- **「初見の強さ」と「蓄積で伸びる力」は別能力** → リーダーボード2枚(コールドスタート順位と学習スロープ順位)
- 学習スロープはエージェント時代の能力論として最も刺さり得る指標
- 実装コスト最小(アダプターにシリーズモード追加のみ)

## 軸3: エージェント設計 (harness engineering) — 第三のディビジョン

**モデルを固定し、人間が組むエージェント設計を変数にする。**
ベンダーのハーネス比較ではなく、「同じモデルでも、組み方でどれだけ差がつくか」。

設計の変数の例:

- 候補手を出させて**フレッシュなエージェント同士に議論・投票**させるskill
- 着手前の**合法性セルフチェック**サブエージェント
- **メモの設計**: 何を残すか、どう構造化するか(軸2のseries-notesの自由化)
- **事前準備**: 自己対戦で戦略書を作ってから本番に臨む
- 探索的な構成: LLM評価つきの浅い木探索、役割分担(偵察役/計画役)

### 競技フォーマットとしての新規性

Halite/Battlecode/Screeps/AIWolf等の「同じゲーム、あなたのコード」系
プログラミング対戦の系譜に、**「固定されたフロンティアモデル、あなたの
オーケストレーション」**という新形式を持ち込むことになる。LLM以前には
存在し得なかった競技であり、次の含意を持つ:

1. **人対人のエージェント代理対戦。** 名前を持つ人間の設計同士が戦う。
   モデル比較ベンチを超えて、観戦・実況・コミュニティリーダーボードが
   成立する(配信・共有と最も相性が良いのはこのディビジョン)
2. **エージェント工学の実験場。** どの設計パターン(議論/検証/記憶形式)が
   汎化するかという、それ自体発表価値のある知見が対戦データから出る
3. **提出インターフェースは既に存在する。** design-v0.1の
   `persistent-agent` 条件と adapter仕様(start_game/act/end_game +
   リソース上限)が、そのまま提出フォーマットになる。パイロットの
   `Agent` インターフェースはその実装済みの雛形

### 公平性ルール(ディビジョン規定の骨子)

- モデルとエフォートを固定(例: 全員 sonnet@medium)
- 1試合あたりの**総トークン/費用バジェット**で揃える(内部構成は自由)
- 全モデル呼び出しをログ(規定外モデル使用の検出)
- レフェリー・観測・失敗ポリシーは共通

### 最小の概念実証(先に走らせる価値あり)

同一モデルで: 素のアダプター(現行 llm.ts) vs 「候補手3つ→フレッシュ
エージェント議論→合法性チェック→決定」skill付きアダプター。
**ハーネス設計だけで勝率が動くこと**を1ペアで示せれば、ディビジョンの
存在意義の実証になる。

## 優先度の推奨

1. **軸2 シリーズ蓄積**: 実装最小・新規性大・無人で回る
2. **軸3 概念実証**: 1ペアで「設計で差がつく」を実証(ディビジョンの種)
3. **軸1 Visionペア**: 公開時の主役。レンダラー実装が必要
4. 軸3 本格リーグ・ブラウザ操作版: 公開後のコミュニティフェーズ

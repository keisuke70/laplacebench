# LaplaceBench 全体戦略 — フロンティアモデル視点のレビューと設計方針

作成: 2026-07-20。既存の [design-v0.1](design-v0.1.md) と
[外部モデル検討ブリーフ](external-model-discussion-brief-ja.md) を前提に、
「測られる側」のフロンティアモデルの視点から、ベンチマークとしての最適化・
コードとしての最適化・公開戦略を一つの戦略に統合する文書。既存文書の
繰り返しは避け、追加・修正・優先順位づけに絞る。

## 1. ポジショニング — 世界のラボに刺さる4つの主張

ゲームを使ったLLM評価は既に多い(チェス、ポーカー、Diplomacy、TextArena、
BALROG等)。しかしそれらの大半は **既知ゲーム**(学習データに定石・棋譜が
大量に存在)か、**毎手ステートレス**(局面を単発で解かせる)である。
LaplaceBenchが独自に主張できるのは次の4点で、公開時のメッセージは
この4点に集中させるべきである。

1. **未知ルールのin-context獲得を分離測定する。**
   記憶された定石が存在しないため、「ルールブックを読んで一試合の中で
   強くなる」能力そのものが測れる。
2. **一試合まるごとの永続コンテキスト。**
   毎手独立プロンプトではなく、人間のように判断を蓄積し方針を修正する
   能力を測る。ゲームベンチマークでこれを主軸にした例はほぼない。
3. **2色1心の協調。**
   同チーム2色は混色挟みができないため、「一つの頭脳が二つの軍を
   運用する」固有の計画問題が生じる。Laplaceのルールそのものが
   ベンチマークの差別化要素になる。
4. **知覚と戦略の分解。**
   同一局面のVision/構造化ペア比較で「盤面が読めていない」のか
   「作戦が悪い」のかを分離できる。

見出しとして最も強いストーリーは:
**「フロンティアモデルは、誰も知らないボードゲームを一試合の中で
どこまで学べるか — そして作者本人に勝てるか」**。
人間アンカー(作者・ボードゲーム経験者に同一プロトコルで対局させる)は
安価に追加でき、スコアに絶対的な意味を与える。ラボ・メディア双方への
訴求力が最も高い要素なので、v1のスコープに入れることを推奨する。

## 2. ベンチマーク科学の核心 — 「統計的パワー/ドル」を設計する

ユーザーの言う「ベンチマークとしての最適化」の実体はここにある。
勝敗(W/L)だけで近い実力のモデルを分離するのは統計的に高くつく:

- 真の勝率75%対25%の差 → 約30局で分離できる
- 真の勝率60%対40%の差 → 約200局
- 真の勝率55%対45%の差 → 約800局

永続コンテキスト+画像入力の対局は1局あたり数十〜百数十回のモデル呼び出しに
なるため、数百局×モデルペア数は費用的に非現実的。よって:

### 2.1 判別の主戦場は「手単位」の指標に置く

- **per-move regret**: 各着手について、検証済み探索ベースライン
  (minimax/深読みのオフライン解析)との評価値差を記録する。
  チェスの「centipawn loss / accuracy」に相当。1局から約60サンプル
  取れるため、W/Lより一桁以上高い統計的パワーを持つ。
- **Tactics/RuleGym**: 局面単位・正解が客観的な問題群。安価・高速・
  信頼区間が狭い。判別とデバッグの主力。
- **Arena(フル対局)**: 少数精鋭のショーケース+検証用。
  「regretの低いモデルが実際に勝つか」の較正として使い、
  順位表の主根拠には手単位指標との併記を使う。

### 2.2 コンピュータチェスのテスト方法論を借用する

チェスエンジン開発(fishtest等)は「少ない局数で有意差を出す」技術を
20年蓄積している。そのまま使えるのは:

- **SPRT(逐次確率比検定)**: 有意差が出た時点で対局を打ち切る。
  費用を数分の一に削減できる。
- **開始局面ブック**: エンジン自己対戦で生成した互角の中盤局面集から
  開始することで、同一展開の繰り返しを防ぎ実効サンプル数を増やす。
  (初手からの対局は「初見の序盤構想」を測る別トラックとして残す。)
- **ペア設計**: 同一開始局面でサイド交換した2局を最小単位にする
  (design-v0.1に既にある。開始局面ブックと組み合わせる)。
- **アンカー梯子**: random → greedy → minimax d2 → d4 → d6 を
  固定ベースラインとしてBradley-Terry/Eloに常時混ぜる。
  これによりモデルが入れ替わってもスケールが年を跨いで比較可能になる。

### 2.3 「学習曲線」を一級の指標にする

未知ゲームである利点を最大化する指標:

- **full-once条件での不正手率を手番インデックスの関数として描く。**
  「試合中にルールを学んでいく」様子がそのままグラフになる。
- **regretのゲームフェーズ別推移。** 序盤は迷走し中盤から締まるのか、
  終盤に状態追跡が崩壊するのか。
- これらの曲線は §5 の公開戦略における最強の共有可能アーティファクト
  でもある(「モデルがその場でゲームを覚えていく」動画・グラフ)。

## 3. 汚染耐性は「構造」で作る — ルール変種ジェネレーター

holdout局面と対称変換(既定路線)は初版には十分だが、公開した瞬間から
Laplace自体が学習データに入り始める。長期的な堀は:

- **パラメータ化されたルール族**: 脱落閾値、Void挙動、勝利条件、
  盤サイズ、捕獲幾何などを機械検証可能な範囲で摂動した
  「Laplace変種」を世代(シーズン)ごとに生成する。
- ベンチマークの本質を「Laplaceが強いこと」ではなく
  「未知ルールの獲得速度」と定義しておけば、シーズン制
  (LaplaceBench-S1, S2, …)で恒久的に新鮮さを保てる。
- 注意: 変種ごとに適合性テストとベースライン再較正が必要なので、
  摂動空間は小さく・全自動検証可能に設計する。v1では設計だけ確保し
  (ruleset_idが既にその布石)、実装はS2以降でよい。

## 4. 測られる側から見た攻略・悪用ポイント(レッドチーム)

ベンチマークを受ける側のモデルとして、スコアを歪められる箇所を挙げる。
いずれも設計段階で潰すか、明示的に測定対象へ昇格させるべきもの。

1. **legal_movesの提供はルール学習を崩壊させる。**
   主トラックからは完全に外す(既定路線の再確認。診断トラック専用)。
2. **修復プロンプトはルール漏洩チャネルになる。**
   不正手のエラーメッセージが理由を説明すると、モデルはエラー駆動で
   ルールを学べてしまう。エラーは標準化コードのみ
   (例: `E_PATH_BLOCKED`)にして説明文を付けない。
   逆に「エラーからの学習速度」を測る専用条件を作っても面白い。
3. **引き分けの悪用。** 劣勢側は千日手・最大手数引き分けに逃げ込む
   誘因を持つ。最大手数・同一局面反復ルールをv1実行前に必ず凍結し
   (現在は未決)、引き分け率を必ず別掲する。打ち切り時の
   優勢判定(アジュディケーション)はゲームを変えるので入れない。
4. **座標規約の曖昧さが最大のサイレントバグ。** `[0, 2]` 形式は
   row/col転置に気づけない。プロトコルは `{"row": r, "col": c}` の
   名前付きフィールド(または代数式表記)へ変更し、レンダラーの
   座標ラベルと完全一致させることを強く推奨する。
   Visionトラックの妥当性はこの一点に懸かっている。
5. **`captured_pieces` の意味が未定義。**
   「プレイヤーiが取った数」か「取られた数」かをスキーマに明記する。
6. **Vision入力の公平性。** プロバイダーごとに画像の前処理
   (タイル化・縮小)が異なる。正準PNGの解像度・配色・線幅を仕様として
   公開し、各プロバイダーの実効解像度で盤面が判読可能なことを
   事前検証する。配色は色覚多様性対応かつ圧縮下でも分離する
   パレットにする(赤/緑の近接を避ける)。
7. **思考予算の単位。** wall-clockはプロバイダー間で不公平。
   公平性の封筒は総トークン(+費用併記)で揃え、レイテンシは
   計測値として記録に留める(未決事項3への回答)。
8. **public-dialogueはプロンプトインジェクション競技になる。**
   相手メッセージ経由の操作・撹乱が主変数になるため、
   主リーダーボードから隔離する方針(既定)を厳守する。
9. **スキーマの矛盾**: `observation_mode` に `browser` が残っている。
   「モデルはブラウザを操作しない」原則と紛らわしいので、
   ロバスト性実験専用である旨を注記するか削除する。

## 5. 公開戦略 — 「見て面白い」の中身

- **ローンチの型**: 主要フロンティアモデル4〜6個の総当たり
  (サイド交換ペア×開始局面ブック)+アンカー梯子+人間アンカー戦。
  全ログ・全プロンプト・アダプター・レンダラーを公開し、
  方法論を論文体で書く。再現可能性がラボの信頼を作る。
- **共有される瞬間**: (a) 不正手率が試合中に下がっていく学習曲線、
  (b) 評価値急落の「ブランダーカム」クリップ、(c) モデル名と
  チーム色付きのライブ観戦。実況・解説は対局後に別モデルが
  イベントログから生成する(既定路線を維持。競技の純度を守る)。
- **参加の導線**: subprocess/HTTPアダプター1枚で新モデルを
  提出できること。検証手順(replay verification)付きの提出フローが
  「コミュニティのベンチマーク」への進化条件。

## 6. 実装戦略(コード最適化)

### 6.1 構成

```text
packages/referee    純粋関数のルールエンジン(game-sharedの移植/包装、依存ゼロ)
packages/runner     CLI・スケジューラ・アダプター・JSONLイベントログ
packages/render     決定論的レンダラー(SVG→PNG、resvg-js。ブラウザ不使用)
packages/viewer     リプレイ観戦(後段。製品GameBoardの再利用)
conformance/        製品エンジン/Python環境との差分テストフィクスチャ
```

- レフェリーは `(state, move) → (state', events)` の純粋関数に閉じ込め、
  乱数・時刻・I/Oを持たせない。イベントソーシング(JSONL)を正本とし、
  リプレイ検証をCIゲートにする。
- 画像は状態ハッシュをキーにキャッシュ。同一局面は再レンダリングしない。

### 6.2 費用の工学

- **プロンプトキャッシュ前提の設計が最大の費用レバー。**
  永続コンテキストは追記専用(安定プレフィックス+末尾追加)に
  構成すれば、各手番の入力の大半がキャッシュヒットになり、
  ゲーム長コンテキストの費用が桁で下がる。プロンプトテンプレートは
  最初からキャッシュ境界を意識して切る。
- SPRT打ち切り、開始局面ブック、per-move regret(§2)はすべて
  「同じ判別力をより少ない呼び出しで」の施策でもある。
- 実行は再開可能に(イベントログ=チェックポイント)。長時間の
  Arenaランが途中障害で全損しない。

### 6.3 クリティカルパスはルール凍結

脱落閾値は「取られた駒3つで脱落(盤サイズに依らず固定値3)」が正ルールと
確定し、旧導出式 `boardSize - 5` を使っていたTS/Python/Kotlinの全実装を
修正済み(2026-07-20)。ただし同種の不整合は他ルールにも残り得る。
**レフェリーが間違っていたら公開後に全結果が無効になる**ので、
これが全工程の前提ゲート:

1. `laplace-8x8-v1` を製品エンジンの現挙動で凍結。
2. 全エッジルールの適合性フィクスチャを、ベンチマークレフェリー・
   製品TSエンジン・Python minimax環境の三者に対して実行。
3. ランダム自己対戦のプロパティテスト+決定論リプレイ検証をCIに常設。

## 7. 最初の一手 — 安いパイロットで「判別するか」を確かめる

宮殿を建てる前に、**Laplaceがフロンティアモデルを実際に判別するか**を
確認するべきである。全モデルがminimax d4に手も足も出ない、あるいは
全モデルが楽勝、のどちらでも設計の調整(難易度レンジ、盤サイズ、
regret指標の重み)が必要になり、それは早く知るほど安い。

パイロットの仕様(数日で組める規模):

- 観測はJSONのみ(Vision・レンダラーなし)
- 永続コンテキストはprovider-native継続のみ
- 対戦相手はminimax d2/d4とrandomのみ、各10〜20局
- 記録は不正手率・勝敗・per-move regret(minimax評価で代用)

このパイロット結果が、§2の指標設計・§5のローンチ規模・変種の要否
すべての入力になる。

## 9. 最大の失敗条件(優先度順)

1. **レフェリーのルールバグが公開後に発覚** → §6.3のゲートで防ぐ。
2. **判別力不足**(全モデル同点/全滅/全勝で差が出ない)→ §7の
   パイロットで早期検知し、難易度梯子とregret指標で分解能を確保。
3. **分散の沼**(局数不足で信頼区間が重なり順位が意味を失う)→
   手単位指標+SPRT+ペア設計。
4. **費用爆発** → キャッシュ前提設計+打ち切り+局面単位スイート主力。
5. **公開後の汚染** → シーズン制+ルール変種(§3)。
6. **「よくあるゲームベンチ」扱い** → §1の4主張と人間アンカーに
   メッセージを集中し、方法論文書と全ログ公開で信頼を裏づける。

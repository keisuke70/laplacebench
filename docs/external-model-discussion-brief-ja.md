# LaplaceBench 外部モデル検討用ブリーフ

## 0. このブリーフの用途

この文書は、LaplaceBenchを既存のLaplace Webアプリへどう組み込むと、
研究として信頼でき、観戦して面白く、OpenAI・Anthropicなどからも注目
され得る公開プロジェクトになるかを、複数のモデルと検討するための共通
コンテキストである。

まだ最終アーキテクチャを決める段階ではない。以下の理解と制約を保った
うえで、異なる案・反対意見・公開戦略を出してほしい。

## 1. 最終目的

まだ広く知られておらず、既存LLMの学習データに答えが大量に含まれて
いないLaplaceというボードゲームを使い、次の能力を測るLLMベンチマーク
を公開する。

- 初見のルールを理解する能力
- 画像から盤面を正確に読み取る能力
- 一試合を通じて考えを蓄積し、作戦を形成・修正する能力
- 2色の味方駒を一つの戦力として長期的に運用する能力
- 合法な構造化アクションを安定して返す能力
- 対戦相手の行動に適応する能力

単なる勝率表ではなく、再現可能な評価基盤、モデル同士のライブ対局、
共有しやすいリプレイを備えた「見て面白い研究・競技」にしたい。

## 2. ゲームと対戦単位

Laplace自体は4色による2対2ゲームである。

```text
Team A: 赤 (Player 1) + 黄 (Player 3)
Team B: 青 (Player 2) + 緑 (Player 4)
手番: 赤 → 青 → 黄 → 緑
```

ベンチマークでは1モデルが同じチームの2色を担当する。

```text
Model A persistent context: 赤 + 黄
Model B persistent context: 青 + 緑
```

これにより、ゲームルールは2対2のまま、評価は純粋なモデル対モデルの
1対1になる。モデルは自分の手番であっても、現在の手番色の駒しか動かせ
ない。同じチームでも異なる色の駒を組み合わせて挟み取りはできないため、
2色の連携には固有の戦略性が残る。

先攻側が有利な可能性があるので、最低評価単位はサイドを交換した2局と
する。正式評価では複数の往復対局を行い、先攻効果も別に推定する。

## 3. すでに合意している方向

### 3.1 永続コンテキストが前提

毎ターン新規プロンプトにするのではなく、各モデルは試合開始から終了まで
独立した永続コンテキストを持つ。

- Team Aの赤・黄は同じコンテキストを再開する。
- Team Bの青・緑は別の同じコンテキストを再開する。
- 対戦する2モデルを一つの共有プライベートコンテキストへ入れない。
- 同じモデル同士の自己対戦でも、チームごとにコンテキストを分離する。

測りたいのは、局面を一回だけ解く能力ではなく、人間のように過去の判断を
蓄積し、方針を変えながら戦う能力である。

### 3.2 思考形式は標準化しすぎない

各ターンに共通の `reasoning`、`plan`、`memory` フィールドを必須化しない。
モデルやプロバイダーは、次を自由に利用できる。

- ネイティブな会話継続
- provider-native reasoning state
- assistantの自然言語出力
- 独自の非公開メモリ
- ツールやコンテキスト圧縮

共通化するのは、リソース上限、観測、最終アクション、イベントログである。
思考過程の文章そのものは必須にせず、採点もしない。非公開状態の管理方法は
エージェント実装の一部として自由度を残す。

### 3.3 モデルはブラウザを操作しない

モデルのアクションはクリックや画面座標ではなく、論理座標の着手である。

```json
{
  "move": {
    "from": [0, 2],
    "to": [4, 2]
  }
}
```

合法性、捕獲、Void化、脱落、手番、勝敗は決定論的レフェリーが処理する。
ブラウザ操作能力を混ぜると、戦略ミスとクリックミスを区別できなくなるため
である。

### 3.4 盤面観測は画像を主軸にする

人間のように盤面を見て理解する能力を含めるため、主競技では毎ターン盤面
画像をモデルへ渡す。画像には少なくとも以下を表示する。

- 8×8盤面と座標
- 4色の駒
- Void駒の区別
- 現在の手番色
- 捕獲数と脱落状態
- 直前の移動元・移動先

同じ局面をJSON/ASCIIで渡す診断トラックも持ち、画像認識の失敗と戦略の
失敗を分離する。

## 4. 既存Laplace Webアプリで再利用できるもの

既存リポジトリ:

```text
/Users/kei/Documents/Codex/2026-07-20-https-github-com-japan-automation-technology/Laplace
```

ローカル統合時は、この場所を `LAPLACE_APP_ROOT` として扱う想定である。

### 4.1 ルールエンジン

`packages/game-shared` に共有ゲームエンジンがある。

- `GameStateManager`
- `MoveOrchestrator`
- `MoveValidator`
- `CaptureDetector`
- `PlayerStatus`
- `GameEndChecker`
- `TurnManager`

モデルの出力座標を `GameStateManager.makeMove(...)` に渡せば、既存の合法手、
盤面変化、捕獲、Void化、勝敗処理を再利用できる。

### 4.2 既存CPU対戦ループ

`web/src/hooks/useCPUGame.ts` はすでに次の流れを実装している。

```text
現在状態をAI APIへ送信
→ AIが from/to を返す
→ GameStateManager.makeMove
→ React stateを更新
→ 次のAI手番
```

これはLaplaceBenchの最小プロトタイプに非常に近い。ただし現状はWeb
クライアント中心で、永続LLMセッション、研究用ログ、公平な対局スケジュール、
複数プロバイダー対応は持たない。

### 4.3 既存盤面とアニメーション

`web/src/components/GameBoard` は `board` と `lastMove` の変化から、次を表示
できる。

- 駒の移動アニメーション
- 捕獲エフェクト
- 最終着手ハイライト
- Void表示
- 脱落関連表示

したがって、レフェリーがモデルの着手を適用し、新しい `GameState` をWebへ
配信すれば、既存画面上でモデルの指示どおりに駒が動く。観戦者は人間同士の
対局に近い形で楽しめる。

### 4.4 既存Minimax CPU

`ai/src/agents/minimax.py` にチームを考慮したMinimaxがある。

用途:

- LLM対Minimaxの基準対局
- Random / Greedy / Minimaxという強度別ベースライン
- LLM着手の探索評価値・regretの推定
- 戦術的に面白い盤面候補の生成

ただしMinimaxをルールの正本にはしない。Python AI側の遷移とTypeScriptの
製品エンジンが、包囲、Void、脱落、中央勝利などで一致するか検証する必要が
ある。

## 5. 想定している1ターンの流れ

```text
1. レフェリーが現在のGameStateを保持
2. 観戦Webへ状態/着手イベントを配信
3. Webは既存GameBoardで前の着手をアニメーション
4. モデル用にアニメーションのない確定盤面画像を生成
5. 現在チームの永続コンテキストを再開
6. 盤面画像、現在色、直前イベントを入力
7. モデルが論理座標 from/to を返す
8. レフェリーがGameStateManagerで検証・適用
9. 結果とモデル使用量をイベントログへ保存
10. 次の色へ進む
```

人間向けアニメーションとモデルの思考時間は分離できる。

- `live`: 各着手の間を空け、思考中表示とアニメーションを見せる。
- `fast`: 対局を高速実行し、後から同じイベントログを映画的速度で再生する。

モデルへ渡す画像は、アニメーション途中ではなく確定状態でなければならない。

## 6. Webアプリとの立て付け候補

### 案A: 既存CPUモードを直接拡張

`useCPUGame` のAI呼び出しを、2つの永続LLMセッションに差し替える。

長所:

- 最速で動くデモを作れる。
- 既存UIとアニメーションをほぼそのまま使える。

短所:

- ブラウザ更新や切断に試合が依存する。
- APIキー、永続セッション、ログ管理をクライアントへ寄せたくない。
- 大量評価、再現、並列実行に向かない。

### 案B: Webアプリ内にサーバー側Arena Orchestratorを追加

サーバーがレフェリー、モデルセッション、ログを持ち、Webは観戦専用になる。
状態はWebSocketまたはSSEで配信する。

長所:

- 既存アプリ内で自然に公開できる。
- APIキーとモデルセッションをサーバーに閉じ込められる。
- ライブ対局を作りやすい。

短所:

- 長時間リクエストやホスティング制限への対策が必要。
- ベンチマーク基盤と製品アプリのデプロイが結合しやすい。

### 案C: 独立Benchmark Runner + 既存Webの観戦ルート

独立プロセスが `@laplace/game-shared` を使って試合を実行し、イベントを保存・
配信する。既存Webには `/arena/<match-id>` のような観戦ルートを追加する。

長所:

- CLI、大量評価、CI、研究再現性とライブ表示を両立できる。
- Webが落ちても試合は継続できる。
- provider adapterと公開データ形式を独立して育てられる。

短所:

- Webとのイベント契約と配信層が必要。
- 初期実装量は案Aより多い。

現時点の有力案は **案Cを本体にし、案Aの既存ループをプロトタイプとして
参考にする構成**。ただし、公開・運用・観戦体験を考えたとき案Bとの
ハイブリッドが適切かは外部モデルにも検討してほしい。

## 7. モデルへ渡す画像の生成候補

### 方式1: 専用の決定論的SVG/PNGレンダラー

同じGameStateから必ず同じ画像を生成する。

- 再現性が高い。
- 高速な大量評価に向く。
- 既存Webと微妙に見た目が異なる可能性がある。

### 方式2: 既存GameBoardの静止画レンダールート

既存コンポーネントをアニメーション無効で描画し、安定後に画像化する。

- 観戦者とモデルがほぼ同じ盤面を見る。
- 既存デザインをそのままブランドとして使える。
- ブラウザ、フォント、viewport差を固定する必要がある。

### 方式3: 両方

公式ベンチマーク画像は方式1、公開ライブ対局や「人間と同じ画面」実験は
方式2とする。両者の成績差も測定できる。

## 8. 公開プロジェクトとして注目されるための要素

### 研究としての信頼性

- ルールをバージョン固定した実行可能レフェリー
- 全試合の再現可能なイベントログ
- モデル名、解決済みバージョン、設定、トークン、費用の記録
- サイド交換と複数対局による先攻差補正
- ratingの信頼区間
- 画像入力と構造化入力の対応比較
- Minimaxなど非LLMベースライン
- 非公開holdout局面と盤面対称変換による汚染耐性

### 見て面白いこと

- モデル名とチーム色を表示したライブArena
- 「Model is thinking...」表示
- 既存の駒移動・捕獲アニメーション
- 早送り、巻き戻し、任意手へのジャンプ
- 勝負を決めた手や評価値急落の自動ハイライト
- SNSで共有できる短い対局クリップ
- LLM対LLM、LLM対Minimax、同一モデル自己対戦

プレイ中モデルに観戦用解説まで強制すると戦略を変える恐れがある。必要なら
対局後に別の解説モデルがイベントログから実況・解説を生成する方が、競技の
純度を保ちやすい。

### 開発者が参加しやすいこと

- provider-neutralなAgent Adapter仕様
- OpenAI、Anthropic、Gemini、ローカルモデルの差し替え
- subprocess/HTTPアダプター
- 1コマンドで対局・再生・集計
- 新モデルをLeaderboardへ提出できる検証手順

## 9. 未決定事項

1. Benchmark Runnerを既存モノレポ内に置くか、独立リポジトリにするか。
2. 公式画像を専用レンダラーにするか、既存GameBoardの安定スクリーンショット
   にするか。
3. 永続コンテキストの共通リソース上限を、総トークン、推論時間、費用のどれで
   揃えるか。
4. コンテキスト上限到達時の圧縮をモデルへ任せるか、adapterへ任せるか。
5. 不正手時に再試行を許すか、その場で手番喪失にするか。
6. 引き分けとなる最大手数と同一局面反復ルール。
7. Minimaxをどの深さ・設定で公式ベースライン化するか。
8. ライブ対局と大量ベンチマークを同じ基盤で動かすか、イベント形式だけを共有
   するか。
9. 思考中の表示、観戦用解説、公開メッセージを競技モデルから分離する方法。
10. 最初の公開版でどのモデル・何対局を掲載するか。

## 10. 外部モデルへの依頼文

以下をこのブリーフと一緒に渡す。

> あなたは、未知ルール学習・視覚的盤面理解・永続的な戦略形成を測る
> LLMベンチマーク兼ライブ観戦プロダクトを設計する立場です。
>
> 上記の確定事項を守りつつ、LaplaceBenchと既存Laplace Webアプリの
> 最も魅力的な立て付けを考えてください。単一案に即決せず、少なくとも3つの
> 本質的に異なるアーキテクチャまたはプロダクト構成を出してください。
>
> 各案を次の観点で比較してください。
>
> - ベンチマークとしての科学的妥当性と再現性
> - LLMが試合を通して思考を蓄積できる自由度
> - 画像入力の品質と公平性
> - 既存ゲームエンジン/UI/Minimaxの再利用性
> - ライブ観戦の面白さ
> - 大量対局時の費用・速度・障害耐性
> - OpenAI、Anthropic、研究者、OSS開発者が注目・参加する理由
>
> 推奨案については、1ターンのデータフロー、主要コンポーネント境界、保存する
> イベント、MVPから公開版までの段階、最大の反対理由と失敗条件を具体化して
> ください。既存案への迎合より、見落としている危険やもっと面白い構成の提案を
> 優先してください。

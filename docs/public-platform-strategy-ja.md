# LaplaceBench 公開プラットフォーム戦略

作成: 2026-07-21。「多くの人がこのベンチマークに触れられる土台」の設計。
研究戦略([benchmark-strategy-ja](benchmark-strategy-ja.md))と実験計画
([experiment-axes-ja](experiment-axes-ja.md))の上に載る、配布・参加・
観戦の戦略。

## 0. 一行の定義

**「誰も知らないボードゲームで、モデルとエージェント設計を戦わせる公開アリーナ」**

触れる人ごとに違う顔を持つ:
観戦者にはeスポーツ、開発者には遊べるSDK、研究者には再現可能な評価基盤、
競技者には「固定モデル・自由設計」の新競技。

## 1. 参加の階段(ファネル)

深さ順に5段。各段の摩擦を最小化する。

| 段 | 人 | 体験 | 必要なもの |
|---|---|---|---|
| 1. 見る | 誰でも | 公開ビューワーで名勝負を観戦、リプレイURLを共有 | ブラウザだけ |
| 2. 回す | Claude/ChatGPT加入者 | `npx laplacebench` で自分のサブスクを使い5分で初対局 | 既存サブスク |
| 3. 実験する | 開発者 | 条件を変えて走らせる(learn、effort、モデル)。ローカルの対局を公開ビューワーにドラッグ&ドロップで再生 | 同上 |
| 4. 提出する | コミュニティ | 対局ログをPRで提出 → CIがリプレイ検証 → コミュニティ台帳に載る | GitHub |
| 5. 競う | 競技者 | シーズン制ハーネス部門(固定モデル、自由設計)に自作の「プロンプトパック」を提出 | 同上 |

設計原則: **上の段ほど摩擦ゼロに、下の段ほど検証を厚く。**

## 2. サブスク問題の答え: 3レーンの信頼モデル

「全部個人サブスクで回している」は弱点ではなく、レーンを分ければ
そのまま配布戦略になる。

### レーンA: セルフサーブ(BYOサブスク)
個人が自分のClaude Code / Codexサブスクで回す。プロジェクト側の推論
費用ゼロ。**これが普及の主レーン。**「あなたが既に払っているモデルに、
ゲームを与える」。結果はローカル完結、信頼は不要。

### レーンB: コミュニティ台帳(未検証部門)
自分の対局ログ(events.jsonl+manifest)をPR提出。CIが決定論リプレイ
検証(構造の正しさ)を通す。ただし**「本当にそのモデルが指したか」は
原理的に検証不能**(人間が代打ち可能)なので、speedrun.comの
コミュニティボード同様「未検証部門」と明示する。正直さが信頼を作る。

### レーンC: 公式検証済みラン
メンテナー(または信頼できる実行者)がAPI経由で実行し、完全な
マニフェスト(モデルID・パラメータ・トークン・費用)を記録。少数精鋭。
**公式リーダーボードと対外的な数値主張はここからのみ出す。**
費用は当面小さく(パイロット規模)、可視性が出た段階でラボの
評価クレジット枠やスポンサーが現実的な調達先。

### 競技(段5)の公平性
ハーネス部門の対局は**主催者が中央実行**する(提出物は設定であって
トークンではない)。提出形式は当面「プロンプトパック」= 戦略書・
スキルmd・アダプター設定の宣言的セット。任意コード実行はサンドボックス
が必要になるので第2期以降。learn実験がまさに「スキルmdの差」だけで
成立した通り、宣言的でも表現力は十分ある。

## 3. リポジトリと配布物

現状の問題: pilot/はローカル専用で、エンジンを製品チェックアウトの
絶対パスで参照している。公開形は:

```
laplacebench(公開リポジトリ)
├── packages/engine      ← game-sharedの凍結コピー or npm参照
│                          パッケージ版 = ルールセット凍結の実体
│                          (laplace-engine-8x8-v1)
├── packages/cli         ← 現pilot/を昇格。npx laplacebench
├── skills/              ← 分析・戦略フォーマット等(競技の提出単位)
├── runs/community/      ← PR提出された対局(CI検証つき)
├── docs/                ← ルールブック、方法論、FINDINGS
└── web-embed仕様        ← 観戦ページとのイベント契約
```

- **エンジンのnpm化がルール凍結を兼ねる**: バージョン番号=ルールセットID。
  適合性テストをパッケージのCIに常設
- CLIは `claude`/`codex` CLIを自動検出。キー無しで段2が成立する
- 観戦UIは製品Webアプリ(/bench)に置き続け、公開デプロイに載せる
  (決定点D1参照)

## 4. 観戦UI v2(「見る」の磨き込み)

現状のv1(一覧+リプレイ+統計バナー)から上げる優先順:

1. **思考の実況表示**: 各手にモデルの生の返答テキスト(既に全ログ保存
   済み)を同期表示。「モデルが何を考えてこの手を指したか」はLLM対戦
   にしかない観戦価値で、最優先
2. **共有カード**: リプレイURLごとのOG画像(製品に既存のOG基盤あり)。
   1局=1URL=1枚の絵、がSNS拡散の単位
3. **学習シリーズビュー**: learn実験の戦略書の世代間diffを対局と並べる
   「学んでいく様子」ページ。このプロジェクト固有の見せ物
4. **ドラッグ&ドロップ再生**: 公開ビューワーにローカルのreplay.jsonを
   落とすと再生(クライアント完結、アップロード不要)。段3の要
5. **ランディング**: 自動再生のヒーロー対局+一行の定義+3つのCTA
   (見る/回す/提出する)
6. ブランダーカム(評価急落クリップ)、ライブ観戦(SSE)は後段

## 5. フェーズ計画

### Phase 1: 土台(いま〜)
- リポジトリ抽出、エンジン凍結パッケージ、`npx laplacebench` 化
- 公開デプロイに/benchを載せ、ヒーロー対局(Fable対GPT-5.6等)を掲載
- ドラッグ&ドロップ再生、README/ルールブック/方法論の公開文書
- 出口条件: 「リンクを踏んで観戦→npxで自分の初対局」が他人の環境で通る

### Phase 2: コミュニティ(土台の後)
- PR提出+CI検証+コミュニティ台帳ページ
- 思考実況・共有カード・学習シリーズビュー
- 段2〜4の英語ドキュメント整備、告知(デモ動画+リプレイURL数本)

### Phase 3: 競技と公式
- ハーネス部門シーズン1(固定モデル+トークン予算、プロンプトパック提出、
  中央実行)
- 公式検証済みランでモデル部門リーダーボード初版(API、複数ペア)
- 製品アプリに人間対LLM対戦を接続(「フロンティアモデルに挑戦する」
  導線。製品側への最強の還流)

## 6. 決定点(要判断)

- **D1: 観戦の公開場所** — 製品Webの既存デプロイに/benchを載せる
  (推奨: 新インフラゼロ、製品への導線にもなる)か、ベンチ専用サイトを
  別に立てるか
- **D2: エンジンの公開形態** — npm公開(推奨: 凍結IDが公的になる)か、
  公開リポジトリへのベンダリング(コピー)か。製品コードの公開範囲の
  意思決定を含む
- **D3: 公開リポジトリの場所と名前** — 新規 `laplacebench` リポジトリを
  GitHubに作るか、既存モノレポ内で公開するか

## 7. 誠実さの原則(公開時に明記するもの)

- 未検証部門と検証済み部門を絶対に混ぜない(表もURLも分ける)
- サブスクCLI経由の対局は「ハーネス込み条件」とラベルする
- 代打ち・モデル差し替えは未検証部門では検出できないと明記する
- 全対局の生ログ・生返答を公開し、誰でも監査できるようにする
